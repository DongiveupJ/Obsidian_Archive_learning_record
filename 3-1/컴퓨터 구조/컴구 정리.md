- • CPU Clocking

    clock: 하나의 기본적인 일을 하는데 걸리는 시간
   ![[Pasted image 20250323221515.png]]
    **Clock period: 클락 사이클의 주기 = 1/frequency**

    **e.g) 250ps = 250x10^-12s - 짧을 수록 좋음**

    **Clock frequency (rate): 초당 사이클 = 1/period**

    **e.g) 4.0GHz - 클 수록 좋음**

**-CPU Time**

    **CPU Time = CC(CPU Clock Cycles)xCCT(Clock Cycle Time)**

        **= CC/Clock Rate**

    **클락 사이클 줄이기, 클락 시간 줄이기로 성능 향상**

    **이 둘은 trade off 관계**

  

**Instruction Count and CPI**

    **Clock Cycles = Instruction Count(IC) x Cycles per Instruction(CPI)**

    **CPU Time = Instruction Count x CPI x Clock Cycle Time**

    **= IC x CPI / Clock Rate**

    **IC는 프로그램을 잘 만드는 것이나, ISA, 컴파일러에 영향을 받음**

    **평균 CPI는 cpu 하드웨어로 줄일 수 있음**

    **Clock Cycles은 각 명령어마다 필요한 클락 수 곱하고 모두 더한 것**

산술 연산
	add +, sub -, 3개의 피연산자 -> 2개의 소스와 1개의 데스티네이션
	
	add a, b, c -> a=b+c 모든 산술연산은 이 모양

레지스터는 32bit 짜리가 32개 있는 형태

산술연산 예제
	f = (g + h) - (i + j);
	->
	add t0, g, h # temp t0(temporary 레지스터) = g + h
	add t1, i, j # temp t1 = i + j
	sub f, t0, t1 # f = t0 - t1 = (g + h) - (i + j)

레지스터 피연산자(operand)
	산술연산은 레지스터 피연산자 사용
	MIPS는 32비트의 레지스터 32개를 가짐 0 to 31
	-> 자주 접근하는 데이터를 레지스터에 넣음
	-> 32비트의 데이터를 "word"라는 단위로 부름
	어셈블러 이름
	$t0, ... , $t9 -> 10개의 임시 값 저장
	$ s0, ... ,$s7 -> 8개의 변수 저장
	개수가 정해진 이유 : 암달의 법칙
	
	디자인 원리 2: 작을수록 빠름 -> 메인메모리 대신 레지스터 사용
레지스터 오퍼랜드 예시
	f = (g + h) - (i + j);
	f, ... , j는 $s0, ... , $s4에 들어가 있음
	add t0, $s1, $s2
	add t0, $s3, $s4. <- 컴파일러가 알아서 넣음
	sub $s0, $t0, $t1

메모리 피연산자
	메인 메모리는 데이터를 결합하기 위해 사용
	->배열, 구조체, 동적 데이터
	2개의 산술 연산 적용
	Load(lw) -> 메모리로부터 레지스터로
	Store(sw) -> 레지스터로부터 메모리로
	==메모리는 byte 단위!!!!!==
	주소 값은 4의 배수(word)로 되어있음
	
	예제1
	g = h + A[8]
	g는 $s1에, h는 $s2에, A의 시작 주소는 $s3에 들어감
	인덱스 8은 32비트 필요
	lw $t0, 32($s3)
	add $s1, $s2, $t0
	
	예제2
	A[12] = h + A[8];
	h -> $s2, A의 시작 주소 -> $s3
	lw $t0, 32($s3)
	add $t0, $s2, $t0
	sw $t0, 48($s3) --- lw와 sw는 방향이 다름! (sw가 다른 연산들의 반대방향)

레지스터 VS 메모리
	레지스터가 메모리에 접근하는 것 보다 빠름
	메모리에 있는 데이터를 다루려면 추가 연산(명령어) 필요 (lw, sw)
	컴파일러는 가능한 레지스터 먼저 사용
	자주 사용하지 않는 변수는 메모리에 저장하여 성능 향상
	-레지스터만 가지고 연산(3개) 하는 것 보다, 레지스터와 상수가 연산되는게 더 빠름(2개)
	-메모리의 데이터를 이용하려면, 그 주소만 알면 됨(데이터는 알 필요 없음)

Immediate Operands -> 상수 연산에 사용
	상수와 레지스터를 연산함
	addi $s3, $s3, 4

	addi $s2, s1, -1

	상수가 작은 경우가 흔하고, Immediate operand가 메모리 로드를 피하기 때문에 빠름

밉스 R 포맷 (R타입) 연산
	![[Pasted image 20250325111446.png]]
	opcode (6비트):
	명령어의 종류를 나타내는 필드로, 64개의 서로 다른 명령어를 표현할 수 있습니다. 6비트는 64 조합을 제공
	rs (5비트):
	첫 번째 소스 레지스터를 지정하는 필드
	rt (5비트):
	두 번째 소스 레지스터를 지정하는 필드로, 역시 5비트로 32개의 레지스터를 표현
	rd (5비트):
	결과를 저장할 목적지 레지스터를 지정. 마찬가지로 5비트로 32개의 레지스터를 지원
	shamt (5비트):
	시프트 연산에서 시프트 양을 나타내는 필드. 시프트할 수 있는 최대 비트 수는 31이므로 5비트가 적합
	-> 최대 31번까지 시프트를 할 수 있는 이유 : 밉스는 32비트 아키텍처로, 각 레지스터는 32비트의 데이터를 저장함. 시프트를 31번 수행하면 모두 0이됨으로, 최대 31번까지 시프트
	funct (6비트):
	특정 연산의 세부적인 종류를 나타내는 필드로, 같은 opcode 내에서 다양한 기능을 구분하기 위해 사용됨. 6비트는 많은 조합을 제공하여 다양한 연산을 지원함
	
	명령어가 32비트라는거지, 명령어가 32개의 레지스터를 모두 사용한다는 뜻은 아님!

프로세서(CPU)
프로세서의 동작 원리를 이해하고, 파이프라이닝을 통해 성능을 최적화하는 방법을 배우는 거야. 데이터 경로와 제어 유닛의 설계, 파이프라인에서 발생하는 위험(Hazards)과 그 해결 방법(포워딩, 스톨, 분기 예측)이 주요 내용이지. 특히, 실제 MIPS 명령어를 예로 들어 각 단계가 어떻게 동작하는지 구체적으로 설명하고 있으니, 예제를 통해 흐름을 익히는 게 중요해.

**명령어 종류와 형식**:
- **R-형 (Register)**: add, sub 같은 산술/논리 연산 명령어. 형식: op(6비트) | rs(5비트) | rt(5비트) | rd(5비트) | shamt(5비트) | funct(6비트).
- **I-형 (Immediate)**: lw, sw, beq 같은 메모리 접근 및 분기 명령어. 형식: op(6비트) | rs(5비트) | rt(5비트) | imm16(16비트).
- **J-형 (Jump)**: 점프 명령어. 형식: op(6비트) | address(26비트).

R타입, Load, Store 데이터흐름
	![[Pasted image 20250514140708.png]]
**(1) R-형 (예: add $rd, $rs, $rt)**
1. **Instruction**: 명령어 입력.
2. **Registers**:
	- rs와 rt 값을 읽음.
	- Write register = rd.
3. **ALUSrc Mux**: 0 (R[rt] 선택).
4. **ALU**: R[rs] + R[rt].
5. **MemtoReg Mux**: 0 (ALU result 선택).
6. **Registers**:
	- RegWrite = 1.
	- ALU 결과를 rd에 저장.
**(2) lw rt, imm16(rs) (예: lw $t0, 20($s0))**
1. **Instruction**: 명령어 입력.
2. **Registers**:
    - rs ($s0) 값을 읽음.
    - Write register = rt ($t0).
3. **Sign-extend**: imm16 (20)을 32비트로 확장.
4. **ALUSrc Mux**: 1 (SignExt(imm16) 선택).
5. **ALU**: R[$s0] + SignExt(20).
6. **Data memory**:
    - MemRead = 1.
    - 계산된 주소에서 데이터 로드.
7. **MemtoReg Mux**: 1 (Read data 선택).
8. **Registers**:
    - RegWrite = 1.
    - 데이터를 $t0에 저장.

 **(3) sw rt, imm16(rs) (예: sw $t0, 20($s0))**
1. **Instruction**: 명령어 입력.
2. **Registers**:
    - rs ($s0) 값을 읽음.
    - rt ($t0) 값을 읽음 (Write data로 전달).
3. **Sign-extend**: imm16 (20)을 32비트로 확장.
4. **ALUSrc Mux**: 1 (SignExt(imm16) 선택).
5. **ALU**: R[$s0] + SignExt(20).
6. **Data memory**:
    - MemWrite = 1.
    - 계산된 주소에 $t0 값을 저장.
7. **Registers**:
    - RegWrite = 0.

밉스 파이프라인
	아래 5단계로 나뉨
	200ps - IF: 메모리로부터 명령어 fetch
	100ps - ID: 명령어 디코드, 레지스터 리드
	200ps - EX: 연산 수행 or 주소 계산
	200ps - MEM: 메모리 operand 접근

파이프라인 datapath/registers

	-> 각 단계가 독립적으로 동작하도록 이전 단계의 결과를 저장하기 위함
	IF/ID: 명령어와 PC + 4를 저장 (32 + 32 = 64비트 = 8바이트)
	ID/EX: 레지스터 값, 부호 확장된 즉시 값, 제어 신호, 타겟 주소를 저장 (32 x 4 = 128비트 = 16바이트)

	MEM/WB: 읽은 데이터 또는 ALU 결과 저장 (32 x 2 = 64비트= 8바이트)

Hazards
	다음 사이클에서 다음 명령어를 시작하지 못하게 하는 상황
	structure hazard -> 하나의 하드웨어에 두 가지 명령 x
	- 자원이 바쁨
	Data hazard
	-이전의 명령어가 데이터 읽기/쓰기를 완료하는 것을 기다려야 함
	Control hazard
	-이전 명령어에 컨트롤 액션을 의존해서 결정하는 것
	

ALU 명령어에서의 데이터 hazards
	다음 명령어가 이전 명령어의 결과를 읽으려 할 때 발생
	주로 ALU 명령어(예: add, sub, and, or)에서 나타남
	sub $2, $1, $3    # $2 = $1 - $3
	and $12, $2, $5   # $12 = $2 & $5 (의존성: $2)
	or $13, $6, $2    # $13 = $6 | $2 (의존성: $2)
	add $14, $2, $2   # $14 = $2 + $2 (의존성: $2)
	sw $15, 100($2)   # Mem[100 + $2] = $15 (의존성: $2)
	-> 데이터 의존 발생 (sub에서의 $2의 결과를 다른 명령어들이 의존)
	파이프라인에서 sub이 $2를 WB(Write Back) 단계에서 갱신하기 전에 and, or, add, sw가 EX(Execute) 단계에서 $2를 읽으려 하면 잘못된 값(예: 이전 $2 값)을 사용하게 됨.
	hazard를 해결하기 위해 forwarding 함

포워딩이 필요한 상황
	1. 파이프라인 레지스터(막대 레지스터)와 레지스터 번호 전달
	EX/MEM 단계의 레지스터와 ID/EX 단계의 rs, rt 번호가 같을 때 포워딩 필요
	-파이프라인 레지스터(막대 레지스터)를 통해 레지스터 번호를 전달
	-ID/EX 레지스터는 현재 명령어의 소스 레지스터(rs) 번호를 저장
	-이는 EX단계에서 ALU의 피연산자로 사용될 레지스터 번호를 제공
	-EX 단계에서 ALU가 사용하는 레지스터 번호는 ID/EX 레지스터의 Rs와 Rt에서 가져옴
	2. 포워딩 조건
	-1a. EX/MEM 레지스터의 타겟 레지스터(Rd)번호가 ID/EX의 소스 레지스터(Rs)와 같음 -> EX/MEM 단계의 결과를 Rs로 포워딩 필요
	-1b. EX/MEM의 Rd가 ID/EX의 rt 번호와 같음 → EX/MEM 단계의 결과를 Rt로 포워딩 필요.
	-2a. MEM/WB레지스터의 Rd가 ID/EX의 Rs 번호와 같음 → MEM/WB 단계의 결과를 Rs로 포워딩 필요.
	-2b. MEM/WB레지스터의 Rd가 ID/EX의 Rt 번호와 같음 → MEM/WB 단계의 결과를 Rt로 포워딩 필요.

포워딩 조건
	EX 해저드
		if (EX/MEM.RegWrite and (EX/MEM.RegisterRd != 0) and (EX/MEM.RegisterRd = ID/EX.RegisterRs))
		-ForwardA = 10
		if (EX/MEM.RegWrite and (EX/MEM.RegisterRd ≠ 0)
		and (EX/MEM.RegisterRd = ID/EX.RegisterRt))
		-ForwardB = 10
	MEM 해저드
		if (MEM/WB.RegWrite and (MEM/WB.RegisterRd ≠ 0)
		and (MEM/WB.RegisterRd = ID/EX.RegisterRs))
		ForwardA = 01
		if (MEM/WB.RegWrite and (MEM/WB.RegisterRd ≠ 0)
		and (MEM/WB.RegisterRd = ID/EX.RegisterRt))
		ForwardB = 01
	RegWrite 활성화, Rd ≠ 0, Rd가 Rs 또는 Rt와 매칭

	동적 브랜치 예측은, Branch prediction buffer (브랜치 히스토리 테이블)를 사용하여 최근 브랜치 명령어의 기록을 저장함.

1비트 predictor: shortcoming
	loop는 거의 무조건 점프함
	내부 루프의 브랜치가 두 번 잘못 예측됨
	Taken(1) -> 분기 발생으로 예측
	Not Taken(0) -> 분기 미발생으로 예측
	마지막 실행 결과에 따라 상태가 업데이트됨
	내부 루프가 여러 번 반복되다가 마지막 반복에서 Not Taken으로 종료되면, 1비트 예측기의 상태가 불안정하게 바뀌어 다음 반복에서 다시 예측 실패가 발생.
	- 내부 루프의 마지막 반복에서 조건이 만족되지 않아 Not Taken이어야 함.
	- 하지만 1비트 예측기는 이전 반복에서 Taken이었으므로 Taken으로 예측.
	- → 잘못된 타겟 주소로 명령어 가져옴 → 플러시(Flush) 필요.
	.
	- 마지막 반복의 예측 실패로 상태가 Not Taken(0)으로 업데이트.
	- 다음 외부 루프에서 내부 루프가 다시 시작될 때(첫 반복), Taken이어야 하지만 Not Taken으로 예측.
	- → 잘못된 Fall-through 경로로 명령어 가져옴 → 다시 플러시 필요.
	-
	- 내부 루프의 마지막 반복(Taken 예측 실패)과 첫 반복(Not Taken 예측 실패)에서 두 번의 misprediction 발생.

브랜치 타겟 계산하기
	예측기를 사용하더라도, 타겟 주소를 계산해야 함
	->1CC 소비
	- 예측기가 Taken으로 예측하더라도, 실제 점프할 타겟 주소(Target Address)를 계산해야 함.
	- 타겟 주소는 명령어 디코딩(ID 단계)에서 즉시 값(Immediate)과 PC(Program Counter)를 사용해 계산됨.
	- 하지만 이 계산은 파이프라인의 IF(Instruction Fetch) 단계 이후에 이루어지므로 지연 발생.
	==브랜치 타겟 버퍼(Branch target buffer)==
	브랜치 명령어의 타겟 주소를 캐싱하는 하드웨어 구조
	->타겟 주소 계산으로 인한 1사이클 페널티를 제거하고, 브랜치 예측과 동시에 올바른 명령어를 바로 패치
	->캐시처럼 동작
	- IF 단계에서 명령어를 페치할 때, PC를 사용해 BTB를 조회
	- 분기 예측에 성공하고 hit(해당 PC가 BTB에 있음) 면, 1사이클 페널티 제거

메모리 계층
	SRAM - static ram, 빠르고 비쌈
	DRAM - SRAM보단 느리고 쌈
	Magnetic disk - DRAM보다 느리지만 제일 쌈
	Ideal memory(이상적인 메모리) - SRAM의 접근 시간, 용량 크고 비용 쌈

지역성의 원칙
	Temporal locality(시간적)
		최근에 사용한 프로그램 올리기
	Spatial locality(공간적)
		가까운 프로그램 올리기

메모리 계층 레벨
	Block: 카피하는 단위 -> word들로 구성
	Hit: 접근하려는 데이터가 상위 레벨에 존재함
	Hit ratio: hits/accesses
	Miss: 블록을 낮은 레벨에서 카피해 다시 가져옴 -> 손해 발생
	Miss ratio: misses/accesses = 1 - hit ratio

캐시 메모리
	CPU에 가까운 레벨의 메모리


Direct Mapped Cache
	메인 메모리 주소로 캐시의 주소 알기
	위치는 주소에 의해 판단됨
	Direct mapped: 하나만 고르기
	![[Pasted image 20250605015740.png]]
	메모리는 32개 블록, 캐시는 8개 블록 -> 8개 블록이므로 3비트 필요
	경우의 수 = 2^n, 뒤의 3비트 = 캐시 주소(어떤 데이터가 들어있는지는 모름)
	tag를 data와 같이 캐시에 저장하여, 태그 2비트로 캐시의 블록 구별
	valid bit로 data의 존재 여부 표시
	1 = 존재, 0 = 없음(초기값)
	캐시 용량 계산 문제

Handling instruction cache miss
	1. PC의 주소(현재 PC - 4)를 메모리로 보냄
	2. 메인 메모리가 읽기 수행, 접근이 끝나기를 기다림
	3. 캐시 엔트리 쓰기:
		- 데이터를 메모리로부터 읽어 캐시의 데이터 필드에 넣음
		- 주소의 상위 비트를 캐시의 태그 필드에 넣음
		- Valid bit 1로
	4. 명령어 실행을 다시함

Write-Through 정책
	데이터가 write될 때 마다 캐시의 블록을 업데이트
	-> 캐시와 메모리의 데이터가 다름
	캐시에서 바뀔때마다 옮김
	캐시에서 업뎃 -> 메모리도 업뎃 해줘야 함

Write-Back 정책
	dirty bit를 사용해 맨 마지막에 바뀐 것만 옮김
	dirty bit: 캐시 블록에 있는 비트, 해당 블록의 데이터가 메인 메모리와 다른지(수정되었는지) 나타냄
	CPU가 캐시에 쓰기 연산을 하면 즉시 메인 메모리에 반영하지 않고, 캐시에만 데이터를 저장
	더티 비트를 1로 설정 -> 나중에 블록 교체 시 메인 메모리에 반영

Virtual Memory
	CPU는 HDD와 직접 연결 되지 않음
	- CPU는 메인메모리를 4GB로 착각
	- 프로그램이 자주 이용하는 코드나 데이터를 메인 메모리에 올림
	- 다른 프로그램엔 접근 못함
	- CPU나 OS가 가상 메모리의 주소(32비트)를 피지컬 주소(30비트)로 바꿔줌
	- 가상 메모리의 블록은 page라고 함
	- 가상 메모리 translation의 미스는 page fault라고 함
	- PM은 VM의 캐시임

Page fault Penalty
	페이지 fault 발생 -> 메인 메모리에 없으니까, 디스크에 접근 해야함
	디스크 접근은 패널티 엄청 큼
	Fully associative placement로 page fault rate 최소화

Page Tables
	Stores placement information
	버츄얼 메모리의 번호가 page table의 인덱스

	만약 페이지가 메모리에 있을 시, 페이지 테이블은 피지컬 페이지의 번호를 저장함
