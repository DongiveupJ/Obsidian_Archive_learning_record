**데이터베이스란?**

**구조화된 데이터의 큰 컬렉션**

**자료구조와는 다름**

**자료구조는 메모리에 저장되지만, 데이터베이스는 외부에 저장됨**

  

**수업에서는 관계형 데이터베이스를 배움**

  

- **•   데이터베이스와 파일시스템의 차이**

    **파일 시스템의 문제점:** 

- **• 중복된 여러개의 데이터를 모두 업데이트 하기 힘듦**

- **• 데이터에 접근하는 것이 어려워 새로운 프로그램을 만들어야 함**

- **• 데이터는 고립과 분리가 필요함 (여러 파일과 형식)**

- **• 무결성 제약조건은 숨겨져있어서 명시되지 않은 코드의 새로운 제약 조건을 추가하거나, 기존 조건을 수정하기 어려움**

- **• 각 데이터의 부분적 수정 등이 어려움**

- **• 동시 접속 시 문제 발생**

- **• 보안 문제**   

    **데이터베이스 시스템은 위 모든 문제를 해결함**

- **• 데이터베이스의 추상화**  

    **사용자들은 내부 구조를 모름**

    **로지컬 레벨: 실제 구조인 피지컬 레벨을 숨기고 더 간소화 함, 일부 정보는 숨김**

**스키마: 구조, 틀 (c++의 타입, 템플릿)**

    **피지컬 스키마: 피지컬 레벨의 데이터베이스 디자인**

    **로지컬 스키마: 로지컬 레벨의 데이터베이스 디자인**

    **피지컬 스키마와 로지컬 스키마가 분리됨 - 서로 영향을 안미침**

    **피지컬 레벨과 로지커컬 레벨을 따로 수정 가능**

    **사용자가 DB한테 로지컬 스키마를 알려주면, DB가 알아서 피지컬 스키마(파일) 생성**

Set 연산
	==union==, ==intersect==, ==except== (자동으로 중복 제거함)
	==union all==
	m + n times in r union all s => 모든 결과를 포함하며, 서로 다른 SELECT 쿼리의 결과가 그대로 합쳐짐
	SELECT column_name FROM table1
	UNION ALL
	SELECT column_name FROM table2;
	=>table 1과 table2에서 가져온 모든 행을 포함함
	==intersect all==
	min(m, n) times in r intersect all s => 두 개의 SELECT 쿼리에서 공통적으로 나타나는 행을 반환. 중복된 행도 포함
	두 SELECT 쿼리에서 동일한 행이 여러 번 나타나는 경우, 그 수만큼 결과에 포함
	SELECT column_name FROM table1
	INTERSECT ALLㅔ
	SELECT column_name FROM table2;
	=> table1과 table2에서 공통적으로 나타나는 모든 행을 포함. 만약 공통된 행이 3번 나타나면 결과에도 3번이 나타남
	==except all==
	max(0, m – n) times in r except all s =>첫 번째 SELECT 쿼리의 결과에서 두 번째 SELECT 쿼리의 결과를 제외. 중복된 행도 포함
	두 SELECT 쿼리에서 동일한 행이 여러 번 나타나는 경우, 첫 번째 쿼리의 해당 행 수에서 두 번째 쿼리의 해당 행 수를 뺀 만큼 결과에 포함
	SELECT column_name FROM table1
	EXCEPT ALL
	SELECT column_name FROM table2;
	이 쿼리는 table1의 결과에서 table2의 결과를 제외한 결과를 반환. 만약 table1에 특정 행이 4번 나타나고, table2에 같은 행이 2번 나타나면 결과에는 그 행이 2번 나타남.

Null 과 3개의 valued logic
	5 < null   or   null <> null   or   null = null ------ unknown으로 나옴
		OR:
		(unknown or true) = true,
		(unknown or false) = unknown
		(unknown or unknown) = unknown
		AND:
		(true and unknown) = unknown,
		(false and unknown) = false,
		(unknown and unknown) = unknown
		NOT:
		(not unknown) = unknown

Referential Integrity(참조 무결성)
	한 테이블의 특정 속성 값이 다른 테이블의 특정 속성 값에 존재해야 함을 보장함 -> 주로 foreign key를 통해 구현
	예: instructor 테이블에서 dept_name이 "Biology"라면, department 테이블에도 "Biology"라는 dept_name이 존재해야 함.
	instructor의 외래 키 = department의 기본 키 = dept_name
	즉, R의 외래 키 값은 S의 기본 키 값과 매핑되어야 함.
	.
	department에서 Biology를 삭제 및 수정하려 할 때, instructor가 Biology를 참조하고 있으면 불가능함
	instructor에서 'dept_name =  physics'를 삽입하려 할 때, department에 physics가 없으면 불가능함
	delete, update는 cascade를 통해 일괄적으로 삭제, 업데이트가 가능
	- ON DELETE SET NULL: department에서 dept_name이 삭제되면, instructor의 dept_name이 NULL로 설정됨.
	- ON UPDATE CASCADE: department에서 dept_name이 수정되면, instructor의 dept_name도 함께 수정됨.
	.
	CASCADE 외에 다른 동작으로 SET NULL과 SET DEFAULT가 가능.
	- SET NULL: 참조된 레코드가 삭제되거나 수정될 때, 외래 키 값을 NULL로 설정.
	- SET DEFAULT: 외래 키 값을 기본값(Default Value)으로 설정.
	.
	==deferred== 키워드로 제약조건 검사를 트랜잭션이 끝날 때(commit 하는 시점)까지 미룰 수 있음

Entity-Relationship Model
	데이터베이스는 이렇게 설계될 수 있다:
		entity의 집합
		entity사이의 관계
	entity는 존재하고, 다른 객체와 구별할 수 있는 객체(예: 특정 인물, 기업, 이벤트)
	entity는 attributes(=열)를 가짐(사람은 이름과 주소를 가짐)
	entity set(=테이블)은 동일한 속성을 공유하는 동일한 유형의 엔티티 집합
	엔티티는 명사(객체), 관계는 동사

Attributes
	entity는 attribute의 집합으로 표현됨
	**Domain**(=데이터 타입) 각 속성이 가질 수 있는 값의 집합
	-> salary 속성의 도메인은 양의 정수의 집합(int)
	-> title 속성의 도메인은 최대 50자 길이의 문자열(char)
	- **Single-valued Attribute (단일 값 속성)**:
	    - 하나의 엔터티가 해당 속성에 대해 하나의 값만 가질 수 있음
	    - 예: ID, name, salary는 단일 값 속성
        - 한 강사는 하나의 ID와 하나의 name만 가짐
	- **Multivalued Attribute (다중 값 속성)**:
	    - 하나의 엔터티가 해당 속성에 대해 여러 값을 가질 수 있음
	    - 예: phone_numbers (전화번호들).
        - 한 강사가 여러 전화번호를 가질 수 있으므로 phone_numbers는 다중 값 속성
    -Derived attribute (파생 속성):
	    다른 속성으로부터 계산될 수 있는 속성
	    -> age = 현재 날짜 - 생일

	one-to-one: 참여 엔티티 집합 중 하나의 기본 키를 선택
	many-to-one: Many쪽 엔티티 집합의 기본 키를 사용
	many-to-many: 참여 엔티티 집합의 기본 키를 모두 조합
	후보 키가 여러 개일 경우, 관계 집합의 의미를 고려하여 기본 키를 선택해야 함

Weak Entity sets
	primary key를 가지지 않은 entity set을 말함
	-> 중복을 제거하는 과정에서 primary key가 삭제됨
	약한 엔티티 집합의 존재는 owner entity set에 의존
	->약한 엔티티 집합은 오너 엔티티 집합과 total, one-to-many관계를 통해 연결되어야 함
	Identifying Relationship은 이중 다이아몬드로 표현
	-약한 엔티티 집합의 식별자(Discriminator or Partial Key)는 약한 엔티티 집합 내의 모든 엔티티를 구분하는 속성 집합임
	-약한 엔티티 집합의 기본 키는 약한 엔티티 집합이 의존하는 강한 엔티티 집합(strong entity set)의 기본 키와 약한 엔티티 집합의 식별자를 조합하여 형성됨
	### 추가 설명
	- **약한 엔티티 집합(Weak Entity Set)**: 독립적으로 존재할 수 없으며, 강한 엔티티 집합(Strong Entity Set)에 의존하는 엔티티 집합입니다. 예를 들어, section이 course에 의존하는 경우 section은 약한 엔티티가 될 수 있습니다.
	- **식별 관계(Identifying Relationship)**: 약한 엔티티와 강한 엔티티를 연결하는 관계로, 약한 엔티티가 존재하기 위해 반드시 필요한 관계입니다.
	- **식별자(Discriminator)**: 약한 엔티티 집합 내에서 엔티티를 구분하는 속성으로, 부분 키(Partial Key)라고도 불립니다.

BCNF와 종속성 보존(preservation)
	-함수 종속성을 포함한 제약 조건은 한 테이블에만 관련된 경우가 아니면 검사비용이 많이 들 수 있음
	-분해된 각 테이블에서 함수 종속성을 테스트하는 것만으로 모든 함수 종속성이 유지되는지 확인할 수 있다면, 그 분해는 종속성 보존(Dependency preserving)이라고 함
	-> 독립적으로 검사를 할 수 있는 속성
	검사를 하기 위해 다시 하나로 합쳐야 하면 Dependency preserving이 아님
	-BCNF와 종속성 보존을 동시에 달성하는 것이 항상 가능하지 않기 때문에, 더 약한 정규형인 Third-normal form 사용 (3NF)

Third Normal Form (3NF)
	BCNF의 조건에서 허용되는 조건이 하나 늘어남(약간의 중복을 허용)
	![[Pasted image 20250417140917.png]]
	위 조건을 하나라도 만족하면 3NF
	+각각의 attribute가 candidate 키에 포함이 되도 ok
	(서로 다른 candidiate 키에 포함되도 괜찮음)
	β – α는 β에서 α에 포함되지 않은 속성들, 이 속성들이 후보 키에 포함되어야 함
	예시
	![[Pasted image 20250417141307.png]]
	학생과 학과의 조합이 many인 many to one
	R = (<u>s_id</u>, <u>dept_name</u>, i_ID)
	f1 - s_ID, dept_name -> i_ID 라는 기능적 의존이 있을 때,
	f2 - i_ID -> dept_name (교수가 하나의 학과만 소속)이라는 제한 추가 -> i_ID에 의해 dept_name이 딸려옴(종속)
	= i_ID는 키가 아니지만, 다른 attirbute를 결정 -> BCNF 위반
	f2로부터 R2 = (<u>i_ID</u>, dept_name)이 나옴
	R1 = (<u>s_ID</u>, <u>i_ID</u>)가 남음
	검사를 하려면 R2, R1 각각 검사가 안됨
	-> R1 ⨝ R2해야 함 -> 종속성 보존x
	종속성 보존을 포기하고 분할을 통해 BCNF로 가거나, 약간의 중복을 허용하고 종속성 보존을 유지하기 위해 3NF로 가거나
	

Normalization의 목적

	R이 "좋은" 폼에 해당하지 않을 때, 분할을 함 {R₁, R₂,...,Rn}
	-분할은 반드시 lossless-join을 지켜야 함
	-항상 만족할 수는 없겠지만, 종속성 보존을 준수하는 것이 더 선호됨(BCNF대신 3NF 사용)

F+(클로저)를 만드는 절차

	![[Pasted image 20250429143725.png]]
	여기서 L은 후보 키가 아닌데 K를 결정함으로 BCNF가 아님
	R을 BCNF로 분해하려 해도 JK -> L이라는 함수 종속성을 보존할 수 없음
	JK -> L을 확인하려면 분해된 관계를 다시 조인해아 하므로, 종속성 보존에 실패
	

마무리(BCNF와 3NF의 비교)
	 **3NF (Third Normal Form) 분해**:
    - 관계를 항상 3NF로 분해 가능.
    - 분해는 **무손실**(lossless)이며, 함수 종속성도 **보존**됨.
	**BCNF (Boyce-Codd Normal Form) 분해**:
    - 관계를 항상 BCNF로 분해 가능.
    - 분해는 **무손실**이지만, 함수 종속성을 **보존하지 못할 수 있음**.
	3NF 분해는 종속성을 보존하지만, BCNF 분해는 종속성 보존을 보장하지 않음 (이전 예시에서처럼 JK→LJK→L 보존 실패).

디자인 목표
	**관계형 데이터베이스 설계 목표**:
    - BCNF를 만족.
    - 무손실 조인(Lossless join).
    - 함수 종속성 보존(Dependency preservation).
	**목표 달성 불가능 시 대안**:
    - 종속성 보존 포기.
    - 3NF 사용으로 중복 허용.
	- SQL의 한계**:
    - SQL은 슈퍼키가 아닌 함수 종속성(FD)을 직접 지정하는 방법을 제공하지 않음.
    - FD를 assertion으로 지정 가능하지만, 테스트 비용이 높고 널리 사용되는 DB에서 지원 안 됨.
    - 종속성 보존 분해를 해도, SQL로 키가 아닌 FD를 효율적으로 테스트하기 어려움.

파일속 레코드들의 구조
	힙
		- 레코드는 파일 내 빈 공간이 있는 곳 어디에나 배치될 수 있음
		- 삽입이 빠르지만, 검색 및 정렬이 느림(빈 공간에 즉시 저장하지만, 특정 레코드 위치를 예측 불가)
		- 순서가 중요하지 않은 데이터에 적합(예: 로그 파일)
	시퀀셜
		- 레코드를 검색 키의 값에 따라 순차적으로 저장
		- 장: 범위 검색 및 순차적 읽기가 빠름
		- 단: 삽입/삭제가 느림
		- 정렬된 데이터가 필요한 경우 사용
	해싱 
		- 각 레코드의 특정 속성(예: ID)에 대해 해시 함수를 적용, 결과값이 해당 레코드가 저장될 블록을 지정
		- 장: 정확한 키 기반 검색이 매우 빠름
		- 단: 해시 충돌 발생 시 성능 저하

시퀀셜 파일 구조
	파일 안의 레코드들이 검색-키에 따라 정렬됨
	전체 파일에서 정렬된 데이터가 필요한 경우 적합
	1. 삭제
		- 포인터 체인을 사용
		- 레코드를 삭제하면 해당 위치를 직접 비우지 않고, 포인터로 연결된 목록(체인)을 통해 삭제된 레코드를 관리 -> 파일의 순차적 구조를 유지하면서 빈 공간을 활용 가능하게 함
		- 삭제된 레코드는 포인터로 연결된 "빈 공간 목록"에 추가됨
	2. 삽입
		- 삽입할 레코드의 적절한 위치를 찾음
		- 자유 공간 있는 경우: 해당 위치에 레코드 삽입
		- 없는 경우: 오버플로우 블록에 레코드 저장
		- 삽입 후 포인터 체인을 갱신하여 순차적 순서를 유지
		- 검색 키 값을 기준으로 정렬된 위치를 찾아 삽입하며, 공간 부족 시 오버플로우 사용
	3. 파일 재조직화 (Reorganiztion)
		- 주기적으로 파일을 재조직화하여 순차적 순서를 복원
		- 삽입/삭제로 인해 순서가 어긋나거나 오버플로우 블록이 증가하면, 파일을 다시 정렬하여 효율성 높임 (피지컬 시퀀스와 로지컬 시퀀스가 동일해야 함)
		- 필요할때 수행

저장공간 접근
	디스크에서 페이지로 가져온 블록, 블록이 꽉 찼을 때 선택해서 다시 디스크로 넘김 - 넘겨지는 블록을 victim(희생자)라고 함
	버퍼: 메인 메모리의 일부, 디스크 블록의 복사본을 저장함, 디스크에서 자주 접근되는 데이터를 빠르게 이용 가능하도록 캐싱, 디스크와 CPU간 데이터 교환의 중간역할을 함

버퍼 매니저
	버퍼 공간을 할당하고 관리하는 서브 시스템 - 어떤 블록이 버퍼에 로드될지, 언제 디스크로 반영될지 결정.
	LRU,MRU 등을 사용해 메모리 효율성을 극대화함
	.
	프로그램이 디스크에서 블록을 필요로 할 때 버퍼 매니저를 호출.
	메인 메모리의 버퍼에서 블록을 관리하여 디스크 입출력을 최소화.
	1. 블록이 이미 버퍼에 있는 경우: 블록이 이미 메인 메모리에 로드되어 있으면, 해당 블록의 메모리 주소를 반환 - 디스크 접근 없이 즉시 사용
	2. 블록이 버퍼에 없는 경우: 공간할당 - 필요시 기존 블록을 교체하여 새 블록을 위한 공간 확보, 교체된 블록은 디스크에 수정된 경우에만 기록
	   블록 읽기 - 디스크에서 요청된 블록을 버퍼에 읽어옴. 블록의 메모리 주소를 요청자에게 반환.
	3. 교체 정책: 버퍼가 가득 차면, LRU를 사용해 덜 사용된 블록 제거
	   수정된 블록만 디스크에 기록하여 불필요한 I/O 방지

버퍼 교체 정책:
	LRU 전략: 가장 최근에 사용되지 않은 블록을 교체 - 쿼리가 잘 정의된 접근 패턴(예: 순차적 스캔)을 가질 경우 예측 실패 가능.
	반복적인 데이터 스캔에서는 비효율적임(중첩 루프 조인 수행 시)
	.
	MRU 전략: 가장 최근에 사용된 블록을 교체
	.
	통계를 기반으로 어떤 블록이 활용될지 확률을 예측해서 전략을 사용하는 것이 중요

Dense Index 파일
![[Pasted image 20250513142857.png]]
	Dense index: 파일에 있는 모든 검색 키(search-key) 값에 대해 인덱스 레코드(index record)가 존재. - 각 레코드마다 개별적으로 인덱스 항목이 생성되므로, 인덱스가 파일의 모든 레코드를 반영

Sparse Index 파일
![[Pasted image 20250513144130.png]]
	Sparse Index: - 파일의 모든 검색 키(search-key) 값에 대해 인덱스 레코드가 있는 것이 아니라, 일부 검색 키 값에만 인덱스 레코드가 포함.
	 모든 레코드를 개별적으로 인덱싱하지 않고, 특정 간격이나 블록 단위로 인덱싱.
	 O(N/B^2) blocks - 블락 당 한개의 인덱스를 저장하면 N/B개의 엔트리 저장, 엔트리를 파일로 저장하면 N/B^2 로 저장 -> dense index보다 1/B만큼 적음
	 블록 안에 있는 데이터를 가져올 때는, 그 데이터만 가져오는게 아니라 블락 단위로 가져오기 때문에, sparse index가 좋은 절충안이 됨
	 sparse 인덱스는 primary 인덱스임 -> secondary이면서 sparse 인덱스일 수는 없음
	 .
	 삽입 삭제 원활하며, 공간이나 비용이 적게듦
	 메모리에서의 비용이 있음 - 블락에 접근을 하고, 블락 안에서 또 찾아야 함 dense인덱스보다 느릴 수 있음
	 블락 하나당 하나의 엔트리가 대표하면 충분
	 .
	 O(log2 N/B)의 메모리 I/O 발생 -> 여전히 비용이 큼

B+ 트리 인덱스 파일
	인덱스드 시퀀셜 파일의 단점
	- 파일이 커질수록 성능 저하 -> 블록이 만들어 질 때마다 많은 오버플로우 발생하기 때문
	- 주기적으로 전체 파일을 필요할 시 reorganization 해줘야 함
	B+트리 인덱스 파일의 장점:
		자동적으로 삽입, 삭제에 따라, 적은 비용으로 스스로 reorganizes 함 
		성능 유지를 요구받지 않고 reorganiaztion 됨
	(거의없음) B+트리의 단점:
		추가 삽입과 삭제 오버헤드, 공간 오버헤드

멀티레벨로 인덱싱을 한번씩 할 때 마다 1/B배씩 사이즈가 작아짐
->업데이트 시 복잡해 B+트리 사용

B+트리의 특징
	- 루트 및 내부 노드: 검색 키를 구분하는 역할, 하위 노드로 이동 경로 제공
	- 리프 노드: 실제 검색 키 값과 데이터 레코드 포인터를 저장 
	- 모든 리프 노드는 연결 리스트로 연결되어 순차적 접근 지원.
	- 검색 키는 리프 노드에만 저장 -> 내부 노드는 경로 안내 역할.
	높이: O(logB N) (삽입과 삭제 cost도 같음)
	리프 노드: 키 - [(B-1)/2, B - 1]개, 포인터 - [(B-1)/2 + 1, B]개
	논리프 노드: 키 - [B/2 - 1, B - 1]개, 포인터 - [B/2, B]개
	루트 노드: 키 - [1, B-1]개, 포인터 - [2, B]개
	리프, 논리프 모두 절반정도 까진 포인터 허용
	한 블록에 들어갈 수 있는 최대 엔트리 수가 B기 때문에, 최대 키의 수는 모두 B-1, 최대 포인터의 수는 모두 B
	.
	블락은 physically close는 아님 -> B+트리에서 바로 다음 노드에 있다고, 실제 블록에서 바로 옆에 있을 필요는 없음(논리적으로만 연결)
	- 레벨이 적어 탐색에 효과적
	- 리프노드에서 split 발생 시, 노드가 위로 올라감(propagate) -> 레벨 증가할 수 있음

B+트리 쿼리
	1. point query
		select * from instructor where ID = '12121';
		인풋: 검색 키 값 v

			C <- root node
			while (C is not a leaf node) {
				C <- C.Pi such that:   (C.Pi는 포인터, 차일드로 내려가는 과정)
					v ∈ Pi's range    
			}
			if (there is C.Ki = v) {
				return the data record referenced by C.Pi
			} else {
				return None
			}
		O(logB N) I/O's
	2. range query
		인풋: 검색 키 값 쌍 [v1, v2]
		아웃풋: <v, ...> 레코드들의 리스트, v ∈ [v1, v2]

			return the leaf node whose range covers v (v를 포함하는 리프노드 리턴)

			C <- find_leaf (v1)
			while (there is C.Ki ∈ [v1, v2]) {
				report all the records C.Pi's such that
				C <- C.next()
			}

Selection Operation
	파일 스캔 방법
	A1 - linear search(sequential scan)시 I/O = O(N/B)
	 메모리가 N보다 더 커도 동일한 코스트
	(조건과는 상관 없음)
	.
	Index 이용 scan(where절에서 if로 조건이 되어있을 때)
	A2 - primary B+tree index, attribute의 키가 한 개일 때(select에서 요구하는 결과값이 1개)
	I/O cost = O(logB N)    =>> B+트리에서 하나 찾기
	A3 - primary B+tree index, 키가 equality 아닐 때(select에서 요구하는 결과값이 여러개)
	I/O cost = O(logB N + T/B)     =>> B+트리에서 range search
	A4 - secondary index, 데이터와 비+트리가 정렬된 기준이 다를 때 -> 데이터는 ID순으로 정렬, 비트리는 department 순으로 정렬됨
	I/O cost = O(logB N + T), T가 더 클 수도 있음
	.
	range search
	A5 - primary index, comparison
	데이터와 b+트리가 A라는 키 기준으로 정렬되어 있을 때
	I/O cost = O(logB N + T/B)
	A6 - secondary index, comparison
	V에 해당하는 것들을 찾고, 엔트리를 시퀀셜 스캔함
	I/O cost = O(logB N + T)
	리프노드를 스캔하면서 데이터를 하나씩 가져옴
	-> 각각의 레코드마다 하나의 I/O 발생 -> 시퀀셜 스캔이 더 나을 수 있음
	T >> N/B

정렬
	M>N이면, 모두 읽어오면 됨
	아니면(메모리가 데이터보다 작으면), external merge sort 해야함
	.
	1. sorted run 생성, i = 0으로 초기화
		M/B(메모리에 들어갈 수 있는 블록 수)만큼 메모리에 읽어옴
		메모리에서 정렬함
		정렬된 데이터를 디스크에 "run Ri"이라는 파일로 저장하고, i증가
		마지막엔 i가 N이 됨
		run의 개수는 최대 O(N/M)개
	O(N/B) I/O's
	2. Merge the runs (K-방향 병합), K < M/B라고 가정함
		1) 메모리에 K개의 런의 첫 블록을 읽어 인풋 버퍼에 저장
		2) 반복함
			1. 정렬된 첫번째 레코드를 모든 인풋 버퍼에서 고름(가장 작은 레코드 선택)
			2. 레코드를 output buffer에 씀, output 버퍼가 꽉차면 디스크에 씀
			3. 인풋 버퍼에서 레코드 삭제. 만약, 버퍼가 비게되면, 해당 런의 다음 블록을 읽음
		3) 모든 입력 버퍼가 비면 종료
		   K < M/B 조건은 병합 시 메모리 충돌 방지
	O(N/B) I/O's
	여러 번 병합 필요 시
		K >= M/B라면, M/B - 1개의 런을 한 번에 병합함
		한 번의 병합으로 런 수를 M/B - 1배 줄이고, 각 런의 크기를 그만큼 늘림
		예: M/B=11, 런 90개면 한 번 병합으로 9개로 줄고, 각 런 크기는 10배.
		반복 병합으로 모든 런을 하나로 통합.
	총 I/O 비용: O((N/B)log⁡M/B(N/B))
