3way handshake를 사용한 연결
	![[Pasted image 20250916175840.png]]
	- 3번의 전송 필요(요청, 요청+응답, 응답)해서 3way
	- TCP는 반드시 이런 셋업 과정을 거쳐야 함
	- 클라이언트-서버 모델: 서버는 항상(먼저) 실행중, 클라이언트가 연결 요청
	- SYN은 연결 요청 패킷임, ACK는 연결 요청 허락
	- 연결은 클라이언트와 서버 양쪽 모두 허가기 필요해서 서버도 SYN을 보냄
	- ACK의 시퀀스# 은 보지 않음
	- 클라이언트에서 보내는 데이터는 8001부터 감
	- 서버에서 보내는 데이터는 15001부터 감
	- 신, 신액, 액 오고감
	![[Pasted image 20250916204533.png]]
	- 서버에서 listen을 호출해야 SYN을 받음(서버가 됨, 버퍼가 생김)
	-  서버에서 accept를 호출하고 블락(클라에서 SYN이 올 때까지)
	- 클라에서 connect를 호출하고 블락(신액이 올 때까지)
	- 클라에서 신액을 받고 connect 리턴(ack 보냄)
	- 서버에서 액을 받고 accept 리턴, read 수행 후 블락(데이터가 올 때까지)
	- 신, 신액, 액은 커널에서 자동으로 보내줌
	- RTO(타이머)를 설정해둬서 응답이 안오면 재전송함(액 등)
	- blocks함수는 호출하기 전까지 이전 단계에서 멈춤
	.
	- SYN 과 SYN + ACK는 데이터를 포함하지 않음, 시퀀스 넘버만 포함
	- ACK은 데이터를 보낼 수 있음(없으면 시퀀스 넘버를 안보냄)

	- 종료되면 양쪽에서 센딩, 리시빙 버퍼 없앰
	- ACK의 seq는 무시
	![[Pasted image 20250917005458.png]]
	- 클라에서 close 호출 시 FIN 전송

	- 서버에서 close 호출 -> FIN 전송 -> 리드랑 같이갈 수도, 따로갈 수도 있음
	- 클라에서 ack 전송하고 종료
	- 종료는 3way or 4way임

	FIN은 시퀀스 넘버 사용
	핀액도 핀의 종류라 시퀀스 넘버 사용
	![[Pasted image 20250917011334.png]]
	- read, write이 클라와 서버에서 동시에 존재하면 작동x -> 블록사용
	- 처음에 서버에서 accept 호출하면 블락
	- 클라에서 connect 호출하여 연결 요청 시 블락상태 -> 서버에서 신액 오면 액 리턴
	- connect가 리턴한 액 받으면 서버에서 accept가 리턴 하면서 read 호출 후 블락
	- 클라에서 write 호출하여 데이터를 보내고 read가 블락
	- 서버에서 데이터를 받고 read가 리턴
	- 서버에서 write로 데이터에 대한 응답을 보내고 read가 블락
	- 클라에서 read가 리턴해서  응답에 대한 액 보냄
	- 클라에서 close로 핀 보냄
	- 서버에서 핀을 받고 리드가 0 리턴하면서 액 보냄
	- 서버에서 close 하면서 핀 보냄 (위 과정과 같이 갈 수 있음)
	- 클라에서 응답으로 액 보내고 종료

TCP 윈도우
	TCP는 데이터 전송의 안정성(흐름 제어, 오류 제어, 혼잡 제어)을 관리하기 위해 윈도우라는 메커니즘 사용
	Send Window, Receive Window
	데이터 전송의 각 방향마다 송신 윈도우와 수신 윈도우가 각각 존재
	양방향 통신에서는 총 네 개의 윈도우가 사용됨
	클라이언트 → 서버 방향: 송신 윈도우 1개, 수신 윈도우 1개
	서버 → 클라이언트 방향: 송신 윈도우 1개, 수신 윈도우 1개

	보내는 데이터의 양이 헤더 크기보다 훨씬 작을 때 -> 비효율적, 낭비

		1. 보낼 데이터가 MSS(Maximum segment size)로 정의된 크기만큼 쌓이면, 상대에게 무조건 보냄
		2. 보낼 데이터가 MSS보다 작을 경우 이전에 보낸 데이터에 대한 ACK가 오기를 기다림. 
		   ACK가 도달하면 보낼 데이터가 MSS보다 작더라도 상대에게 보냄
		Nagle은 디폴트로 동작(어플에 따라 다름)
	- 수신 측에서 발생하는 신드롬

		2. 확인응답의 지연: 버퍼에 일정 크기 이상의 빈 공간이 생겼을 때 액 보냄(액 전송 지연) -> 조금 더 보낼 수는 있지만, ACK이 오지 않아 결국 stop (sender가 받은 윈도우 사이즈까지만 데이터를 보내는데, ACK이 안와서 stop)
		리시버 어플리케이션이 버퍼에서 너무 천천히/조금씩 가져가서 발생

TCP 혼잡 제어 요약
	![[Pasted image 20251015124305.png]]
	처음엔 slow start로 시작 -> threshold까지 cwnd 2배씩 exponential하게 증가
	스레시홀드까지 도달하면 congestion avoidance로 1씩 additive하게 증가
	패킷 손실 감지 시 줄임
	Congestion avoidance 시
	1. 3duplicate ACKs: 혼잡 감지(패킷 손실)시 윈도우 사이즈를 절반으로 줄임
	2. Time-out: 모든 패킷 손실, 윈도우 사이즈 1MSS부터 다시 시작, 스레시홀드 해당 cwnd의 절반으로 수정
	Slow start시
	3. 3duplicate ACKs: CA와 동일 -> 윈도우 사이즈 절반
	4. Time-out: CA와 동일 -> 윈도우 사이즈 1, 스레시홀드를 time-out 발생했던 윈도우 사이즈의 절반으로 수정

Smoothed RTT
	EWMA 방식으로 구함(몰라도 됨)
	처음엔 아무 값도 설정 안됨
	첫 번째 측정 이후 => RTTs=RTTm
	그 이후의 측정 후=> RTTs = (1-a)RTTs + a RTTm
		                = 7/8 x 기존의 평균 + 1/8 x 지금 측정한 값
	가중치가 다름
RTT Deviation(편차)
	처음 => 아무 값도 설정 안됨
	첫 번째 측정 이후 => RTTd= RTTm/2
	그 이후의 측정 후 => RTTd = (1-b)RTTd + b|RTTs-RTTm| = 3/4 x 기존편차 + 1/4 x 지금 구한 편차
RTO
	처음 => 초기값
	그 이후의 측정 후 => RTO = RTTs + 4xRTTd
	![[Pasted image 20251029160340.png]]
	RTO는 처음엔 디폴트 값(6초)이며 의미없음
	첫 번째 측정 -> M = S = 1.5, D = 0.75(1.5 / 2)
	![[Pasted image 20251029162359.png]]
	두 번째 측정 -> M = 2.5, S = 1.5x7/8 + 2.5x1/8 = 1.625, D = 3/4x0.75 + 1/4x|1.625-2.5| = 0.78, RTO = 1.625 + 4x0.78 = 4.74
	![[Pasted image 20251029162428.png]]

Karn's 알고리즘
	재전송하는 모든 패킷에 대해서 RTT측정 및 업데이트 x
	![[Pasted image 20251029163309.png]]
	이유 -> 모호성 방지
		송신자가 여러 패킷을 연속으로 보내고 ACK를 기다리는 상황
		- 송신자가 패킷 X를 보낼 때 타이머 시작
		- 패킷 X가 손실되어 액이 오지않고, RTO가 만료되기 전에, 송신자는 패킷 X를 재전송
		- 이후 수신자로부터 패킷 X에 대한 액 도착
		이때 송신자는 다음과 같은 모호성에 직면함
		- 이 액이 1차 전송된 패킷 X에 대한 응답인가?
		- 아니면 2채 재전송된 패킷 X에 대한 응답인가?
		이러한 모호성 때문에 재전송된 패킷의 ACK 시간을 RTT측정에 사용하면 RTO 값이 부정확해질 수 있음 -> Karn의 알고리즘으로 해결
		결과적으로, 송신자는 최초로 전송된 후 아직 액을 받지 못한 패킷에 대해서만 RTT측정 진행 -> RTT 측정이 한 번에 유일한 샘플만 진행되도록 제한
	![[Pasted image 20251029164737.png]]
	Exponential Backoff: 재전송 시 RTO를 2배 늘려서 재전송
	재전송 시 걸린 RTO는 Karn의 알고리즘에 의해 반영이 안되는 모습
	트래픽에 따라 RTT 변화 큼 (거의 두배가까이 차이남)

Netid & Hostid

	Netid - 네트워크를 대표하는 id(기지국)

	1번째 바이트로 클래스 판별 후 Netid 파악하기
	Class A
		Netid: 0~127
		hostid: 0.0.0 ~ 255.255.255 (1600만개 = 2^24)

	Class B

		B도 많이 낭비됨
	Class C
		Netid: 192.0.0 ~ 223.255.255
		hostid: 0 ~ 255 (256개 = 2^8)
		C는 많이 적음
	A, B, C는 2^8만큼 차이남
	Class D
		클래스 D는 멀티캐스트용으로 쓰임
		multicast: IP주소, 주소를 그룹화한 1:n 통신
		IPTV에서 주로 쓰임
		(unicast - 1:1, Broadcast - 1:all)
	Class E
		나중을 위해서 남겨둔 주소(안씀)
	수업에서는 A, B, C만 다룸

# 7장 IP(Internet protocol)

IPv4다룸
IP는 네트워크 레이어 -> 패킷: datagram

IP datagram
	![[Pasted image 20251125101750.png]]
	VER: 버전
	HLEN: 헤더 길이
	Service type: 8비트임, 용도는 넘어감
	Total length: IP헤더를 포함한 데이터그램의 길이. 46보다 작을 시 패딩 들어감
	Time to live: 라우터 지날 때마다 -1, 0되면 패킷 소멸 -> 라우터를 몇번 지날 수 있는지
	Protocol: 프로토콜 구분
		ICMP, IGMP, TCP, UDP, OSPF

작은 데이터그램의 Encapsulation
	![[Pasted image 20251125102713.png]]
	데이터그램의 minumum크기는 46바이트 -> 부족할 시 패딩으로 채워줌
	IP, TCP헤더의 크기가 최소 20바이트 -> 40바이트, 데이터가 6바이트 미만일 때 패딩
	CRC -> 비트 에러 찾기, 체크섬이랑 유사함

예제
	도착한 IP패킷의 처음 8비트가 0100 0010일 때 리시버가 버린 이유?
	뒤 4비트가 HLEN -> 0010 = 2, 2x4 = 8 헤더의 길이가 8비트임
	-> 헤더 길이가 8바이트여서 버림 (헤더의 최소 길이는 20바이트)
	.
	IP패킷에서 HLEN의 값이 이진수 1000 일때, 옵션이 몇바이트 있나?
	1000 = 8, 8x4=32 헤더의 길이가 32바이트
	32 - 20 = 12바이트
	옵션은 12바이트 사용
	.
	total length - HLEN = 데이터크기

Fragmentation
	각 피지컬 네트워크 마다 보낼 수 있는 맥시멈 크기가 있음 = MTU
	-> 피지컬 네트워크가 다르면 크기 달라짐 -> 분할 필요

MTU
	![[Pasted image 20251125104313.png]]
	MTU: 데이터그램의 최대 크기(헤더 + 데이터)
	min: 46, max:1500(이더넷)
	헤더는 그대로, 데이터만 분할됨
	분할된 데이터마다 헤더는 그대로 붙음

Flags field
	![[Pasted image 20251125104554.png]]
	D: Do not fragment, 1이면 분할x -> 크면 패킷 버림
	M: More fragments 재조합 시 필요

Fragmentation example
	![[Pasted image 20251125105203.png]]
	fragment offset은 13비트임 -> 16비트 표현 못함
	2^16 / 2^13 = 8(3비트)로 나눠줘서 fragment offset 필드에 저장
	![[Pasted image 20251125110516.png]]
	identification필드: 조합하기 위해 사용 -> 같은것끼리 조합
	MTU: 1420
	more fragment: 뒤에 쪼개진 패킷이 더 있을 시 1, 없으면 0(마지막)
	쪼개진 패킷은 경로면에서 독립적임
	.
	M비트가 0이면 몇번째 프래그먼트? -> 쪼개진 것의 마지막 패킷이거나 안쪼개진 패킷
	M비트가 1이면? -> 마지막이 아닌것만 알 수 있음 -> 처음인지 중간인지 모름
	M비트가 1이고 fragmentation offset은 0임 -> 쪼개진 패킷의 첫번째
	오프셋 값이 100일 때, 첫번째 바이트는 100x8=800, 마지막 바이트는 모름
	오프셋이 100이고, HLEN은 5, total length는 100일 때, 첫번째 바이트와 마지막 바이트?
	-> 첫번째 바이트=800, 헤더길이는 5x4=20바이트, 총길이는 100바이트니까 100-20=80, 800 + 80 = 879 -> 마지막 바이트

	- 목적지와의 통신이 관리적으로 금지됨: code13 - 어떤 이유든지 목적지가 통신을 원하지 않을 경우 -> 방화벽 등

2-node instability
	![[Pasted image 20251202163722.png]]
	두 개의 노드간에 루프 발생하는 것
	X와의 연결이 끊어진 상태에서, A가 B에게 정보를 주기 전에 B가 A에게 줌
	-> loop 발생
	X로 가는 패킷이 하나 들어오면, cost가 맥시멈까지 상승해 infinity가 되어버림
	.
	해결방법
	1. infinity = 16 으로 assign (16은 AS에서 큰 숫자)
	2. Split horizon: B에서 next가 A면 A에게 정보 전달x
	   next가 상대방이면 그 정보만 안보냄 단점: 안보내는 이유가 일부러인지 몰라서인지 모름
	3. Split horizon & poison reverse
	   next가 상대방이면 cost를 infinity로 보내줌
	위의 방법으로도 모든걸 해결하진 못함

3-node instability
	![[Pasted image 20251202173227.png]]
	split horizon / + poison reserve로도 해결 안됨
	-> 2-node instability로 해결이 안되는 상황
	- A -> B, C인 상태
	- X가 끊기면 A가 B, C로 정보를 보내줘야 하는데, A->C 패킷이 사라짐
	- 이떄, C->B로 보내주고, B는 최소 코스트이기 때문에 업데이트 함 -> 문제발생
	  2-node instablilty 떄문에 B->A로는 안갔음
	- next가 달라져서(A가 아닌 C로)B->A로 보내고 A가 없데이트 -> loop발생
	해결 방법: 껐다 키기 (감지 및 해결 어려움)

# 8장 ARP

address resolution protocol
logical address(IP)와 physical address(MAC)를 매핑하는 프로토콜


![[Pasted image 20251205234719.png]]
특정 IP주소에 해당하는 물리적 주소(MAC)를 알아내는 것이 ARP의 목적

ARP operation
	![[Pasted image 20251205234811.png]]
	먼저 브로드캐스트로 보낸 후, IP가 일치하는 dest에서 소스로 응답을 보내줌(dest의 맥주소를 모르는 상황)
	-> 소스에서 해당 IP의 MAC주소를 알게됨

ARP 패킷
	![[Pasted image 20251205234919.png]]
	Hardware Type: 하드웨어 주소(MAC)의 유형을 나타내며, 이더넷 통신시 항상 1로 설정
	Protocol Type: 매핑 대상인 프로토콜 주소의 유형을 나타내며 IPv4의 경우 0x0800 설정
	Hardware Address Length: 하드웨어의 길이, Byte로 표시, 이더넷 상에서는 0x06으로 설정
	Protocol Address Length: 프로토콜 주소 길이, Byte로 표시, IPv4의 경우 0x04로 설정
	Operation Code: ARP의 구체적인 동작을 나타냄
		ARP: IP주소 -> 맥주소
		RARP: MAC주소 -> IP주소
	.
	ARP패킷의 캡슐화
	![[Pasted image 20251205235246.png]]
	Preamble - 수신지의 동기화를 위해 사용
	Destination Address - 프레임 송신 6바이트
	Source Address - 프레임 수신 6바이트
	Type - 상위계층 프로토콜 정보(IP=0x0800, ARP=0x0806) 2바이트
	Data - 상위계층으로 부터 전달받은 데이터, 최소 46바이트, 최대 1500바이트
	상대방 주소를 알 수 없어 데스티네이션 어드레스에 브로드캐스트 주소를 씀

ARP 리퀘스트는 브로드 캐스트지만, ARP 리플라이는 유니캐스트임

ARP의 4가지 케이스
	![[Pasted image 20251206001141.png]]
	1. 센더와 리시버가 같은 네트워크(host -> host) 
	2. 센더가 다른 네트워크로 보낼 때(host -> router)
	3. 라우터가 다른 네트워크로 보낼 때(router -> router)
	4. 라우터가 같은 네트워크로 보낼 때(router -> host)

예제
	![[Pasted image 20251206001521.png]]
	리퀘스트 보낼 때는 dest를 브로드캐스트로, dest의 MAC주소는 모르니 0으로
	리플라이 보낼 때는 받은 리퀘스트의 주소를 목적지로

ARP components
	![[Pasted image 20251206002128.png]]
	캐시-컨트롤 모듈
		60초마다 깨어나서 테이블 점검함
		- P일 때 attempt를 1 증가시키고 리퀘스트 다시 보내거나, attempt가 맥시멈보다 커지면 F로 바꾸고 큐 지움
		- R일 때, time-out을 60 깎음, 0 이하면 F로 바꾸고 테이블 지운 후 큐를 삭제함
	인풋 모듈
		1. 엔트리를 업데이트 함
			1. P였으면 R로 바꿔주고 MAC주소를 엔트리에 기입 
			2. 이미 R이면 time-out 리셋
		2. 받은게 리플라이 일 때 - Dest로부터 응답이 왔을 때, 엔트리가 pending이면 큐에 있던 패킷들 모두 전송해줌(데이터 링크 레이어로)
		3. 처음 ARP패킷(최초 리퀘스트)를 받았을 때(엔트리에 없을 때)
			1. ACK을 보내기 위해 테이블에 소스의 주소 저장 필요
		4. 받은게 리퀘스트일 때, 응답 보내줌
	아웃풋 모듈
		 - R이면 패킷 보냄
		 - P면 큐에 집어넣음
		 - 엔트리가 없으면 캐시 테이블에 엔트리 한 줄 생성 후, Pending, attempt = 1로 만들고 큐를 생성하여 보낼 패킷들을 큐에 저장함(첫번쨰 이후의 패킷들)
