# 1.Introduction
Motivational Example
	인풋: n개 숫자의 Array A
	아웃풋: A의 원소들의 합
	과제: 이걸 병렬로 처리
	.
	Assumptions:
		계산: 2개 숫자 더하기 1사이클
		커뮤니케이션: 데이터를 다른 메모리로 옮기는데 3사이클
		인풋, 아웃풋: PE #0 에서 모든 데이터 시작 및 종료
		동기화: 모든 PE는 같은 동작 수행(같은 계산 or 소통)
	T(n, p)
	n: 처리할 양, p: 프로세서 수
	T(n, 1) = n-1사이클 -> 프로세스 1개일 때 n-1사이클 소요
	Speedup: S = T(n, 1) / T(n, p) 1개 대비 P개 사용시간
	Efficiency: E = S / p 사용한 프로세서 대비 얻은 속도 증가 효과
	Cost: C = T(n, p) x p 총 비용, 1개로 100초 = 100개로 1초 ->cost는 같음
	Scalability: 확장성, 프로세서의 수가 변할 때, 수행시간이 얼마나 리니어하게 변하는지?

PE가 2개일 때(p = 2, n = 1024)
	T(1024, 2) = 3(보내기) + 511 + 3(모아오기) + 1(합치기) = 518사이클
	S = 1023 / 518 = 1.975
	E = 1.975/2 = 98.75%

PE 4개일 때 (p = 4)
	T(1024, 4) = 3x2 + 255 + 3x2 + 2 = 269사이클
	S = 1023/269 = 3.803
	E = 3.803/4 = 95.07%

PE 8개일 때
	T(1024, 8) = 3x3 + 127 + 3x3 + 3 = 148사이클
	S = 1023/148 = 6.91
	E = 6.91/8 = 86%
	->프로세스가 늘어날 수록 효율 줄어듦(communication 비용 증가)

예시 통계
	p = 2^q, n = 2^k
	데이터 분배: 3 x q
	각 합 연산: n/p - 1 = 2^k-q - 1
	각 결과 모으기: 3 x q
	합치고 더하기: q
	T(n, p) = T(2^k, 2^q) = 3q + 2^k-q - 1 + 3q + q = 2^k-q - 1 + 7q
	나누고 합치는 비용이 프로세서의 수에 비례해 커짐
	Strong Scalability (문제 사이즈 고정)에서는 프로세서의 수가 커질수록 무조건 좋은게 아님
	Weak Scalability (문제 사이즈도 같이 증가)에서는 프로세서의 수가 커질수록 속도, 효율 계속 증가
	-> 병렬 프로그래밍은 weak scalability가 좋음

# 2.Theoretical Background

PRAM(Parallel Random Access Machine)
	여러개의 프로세서들이 공통으로 공유되는 메모리에 접근함
	UMA임
	프로세서들끼리 직접적인 소통은 불가->공유메모리에 접근해서 간접적으로 소통
	모든 프로세서는 동시에 같은 작업 수행
	아래 3개 페이지로 나뉨
	1. Read phase: 메모리에서 레지스터로 읽어옴
	2. Compute phase: 레지스터의 로컬 데이터로 연산 후 레지스터에 저장
	3. Write phase: 데이터를 공유메모리 셀에 저장, 여러개의 프로세서들이 동시에 같은 주소에 쓰는 것을 막기 위해 EW/CW사용
	각 스텝은 상수시간
	.
	PRAM의 종류
	- Exclusive Read Exclusive Write(EREW): 같은 주소에서 두 개 이상의 프로세서가 동시에 접근 못함
	- Concurrent Read Exclusive Write(CREW): read는 동시에 가능, write는 동시에x -> 가장 일반적임
	- Concurrent Read Concurrent Write(CRCW): write하는 방식이 다름
		1. priority CW: write에 우선순위 정함
		2. Arbitrary CW: 프로세서를 랜덤하게 골라 write함
		3. common CW: 모든 프로세서가 공통된 값을 쓰려고 하면 씀, 그렇지 않으면 x
		4. combining CW: 여러개의 값을 조합해서 씀(합, 곱, 최솟값, and연산 등)
	EREW, CREW, CRCW 3가지 모델 주로 사용

Parallel Prefix Computation
	Binary associative operation 
	- 결합법칙이 성립함 (xi​∘xj​)∘xk​=xi​∘(xj​∘xk​)
	- 프리픽스 연산의 정의
		- 입력 배열 **X={x0​,x1​,...,xn−1​}** 가 주어졌을 때,
		- 출력 배열 **S={s0​,s1​,...,sn−1​}** 을 계산하는 것을 의미
		- si​=x0​∘x1​∘...∘xi​
		- 즉, 각 si​는 입력 배열의 첫 번째 원소부터 xi​까지 주어진 연산을 누적한 결과
	최소한의 시간 복잡도 Ω(n)
	**병렬 알고리즘**의 경우, 더 빠른 시간 복잡도로 계산할 수 있지만, 아무리 빨라도 입력 크기 n에 비례하는 시간보다는 더 적게 걸릴 수는 없다는 의미

Foster's Parallel Algorithm
	효율적인 병렬 프로그램을 만들기 위해 문제를 체계적으로 분석하고 설계하는 4단계의 접근법
	1. 파티셔닝
		전체 문제를 여러 개의 세분화된(fine-grained) 작은 작업들로 나눔 -> 가장 작은 단위로 나누기
		가능한 많은 독립적인 작업들을 식별하여 병렬 실행의 기회를 극대화 함
		나누는 방법 여러가지 있음 (1장에 나옴)
	2. 통신(communication)
		파티셔닝 단계에서 나누어진 작업들 간에 필요한 데이터 교환의 종류와 양을 결정
		각 작업이 다른 작업의 결과를 필요로 하는지, 얼마나 자주 통신해야 하는지 등을 파악해서 통신 비용을 예측
	3. 집합화 (Agglomeration)
		통신 단계에서 식별된 작업들을 더 큰 덩어리(coarse-grained)의 작업으로 묶음
		통신 오버헤드를 줄이고, 데이터 지역성(data locality)을 개선하여 전체적인 효율성을 높임 -> 너무 세분화된 작업은 통신 비용이 커질 수 있으므로, 적절한 크기로 묶는 과정 필요
	4. 매핑
		집합화된 작업들을 실제 프로세서나 스레드에 할당
		통신을 최소화하고, 작업 부하를 균등하게 분배하여 모든 프로세서가 효율적으로 작동하도록 만듦 -> 이 과정에서 동시성(concurrency)을 확보하는 것도 중요

	![[Pasted image 20251019201220.png]]
	 공유변수를 쓰기 위한 필수요소:
		 conditional variable (신호) -> cv
		 상태(공유변수) -> 여기선 condition
		 Mutex -> 공유변수에 대한 접근 제어
		1. unique_lock으로 잠그기
		2. 상태 확인 -> `if(?condition) {cv.wait(unique_lock)}` -> wait 하면서 열기
	 
	 **`cv.wait()`의 3단계 동작 원리**
	 `cv.wait(lock);` 코드가 실행될 때, 내부적으로는 다음의 세 가지 작업이 **원자적(Atomic)**으로, 즉 중간에 다른 스레드가 끼어들 수 없도록 안전하게 일어납니다.
	
	1. 뮤텍스 해제 (Unlock): `wait()` 함수는 인수로 받은 `std::unique_lock` 객체(뮤텍스를 소유하고 있음)를 **해제(unlock)**합니다.
	    
	2. **스레드 대기 (Sleep)**: 락을 해제한 직후, 현재 스레드는 조건 변수에서 **잠이 듭니다 (블로킹 상태)**.
	    
	3. **뮤텍스 재획득 (Relock)**: 다른 스레드에서 `notify_one()` 또는 `notify_all()`이 호출되어 현재 스레드가 깨어나면, 스레드는 임계 구역을 벗어나기 전에 **뮤텍스를 다시 잠급니다 (relock)**.
	    
	
	### **왜 Unlocking이 필요한가요?**
	
	`wait()`가 락을 해제하는 이유는 **경쟁 상태(Race Condition)를 방지하고 동기화 로직을 완성**하기 위함입니다.
	
	- 만약 `wait()`를 호출할 때 락이 해제되지 않으면, **알림을 보낼 스레드(Notifier)**는 공유 데이터(`condition`등)를 수정하고 `notify_one()`을 호출하기 위해 **임계 구역(Critical Section)에 진입할 수 없습니다.**
	    
	- 락을 해제함으로써, 기다리는 스레드(Waiter)는 잠들고, 알림을 보낼 스레드가 락을 획득하여 조건을 `true`로 변경한 후 알림을 보낼 수 있게 됩니다.
	    
	따라서 `cv.wait()`는 "내가 잠들 테니, 뮤텍스를 놓아줄게. 조건을 바꾼 다음 나를 깨워줘"라는 복합적인 작업을 안전하게 수행하는 메커니즘입니다. 이것이 `std::unique_lock`만이 `cv.wait()`의 인자로 사용될 수 있는 이유이기도 합니다 (락을 수동으로 해제하는 기능이 필요하기 때문입니다).
.

	cv.wait은 인자로 반드시 유니크락 필요 -> wait시에는 lock을 풀어야 하기 때문에 유니크락 사용
	->락가드는 중간에 lock을 풀지 못함
	`while (!condition) cv.wait(unique_lock)` = `cv.wait(unique_lock, [&](){return condition})`

Reduction operator
	![[Pasted image 20251111002650.png]]
	reduction operator인 x를 항등원으로 초기화 해야 함(덧셈:0, 곱셈:1) -> 연산마다 달라짐
	스레드마다 할 일 수행 후 밖의 공유 변수에도 계산해줌
	![[Pasted image 20251111094336.png]]
	- x는 초기값 5를 가짐
	- 2개의 스레드를 이용해 i루프 병렬화 -> 스레드당 5개씩 나눠가짐
	- `reduction(min:x)`각 스레드의 로컬변수 x에 대해 계산을 수행 후, 최종적으로 전역변수 x와 모든 로컬 x값 중 가장 작은 값을 선택
	- x를 해당 타입의 가장 큰 값으로 초기화(2^64 - 1)
	- 루프 내에서 덧셈 수행 -> 오버플로우 발생
	- 5번의 덧셈 결과, 로컬x는 5가 아닌 4가 됨
	- OpenMP는 전역변수와 모든 로컬 결과(5, 4, 4)중에서 최솟값을 찾게됨
	- 4가 출력
	-> 연산자에 따라 로컬 변수의 초기값이 다르게 설정되며, 오버플로우를 조심해야 함

사용자 정의 리덕션
	리덕션 연산이 병렬로 안전하게 수행되기 위해서는 다음 세가지 수학적 속성을 만족해야 함
	1. 결합 법칙
	2. 항등원: 연산에 영향을 주지 않는 특별한 초기값(e)이 존재해야 함 -> 각 스레드의 로컬 리덕션 변수의 초기값으로 사용
	   절대값 최대(absmax)의 항등원은 0 absmax({v, 0})
	3. 교환 법칙
	.
	`#pragma omp declare reduction(identifire : type : combiner) 
	`initializer(omp_priv = <init-value>)`
	identifier: 연산의 이름(custom reduction)
	type: 데이터 타입
	combiner: 로컬 결과를 표현
		omp_out = `<operation using omp_out and omp_in>`
		ex) omp_out = omp_out + omp_in
	initializer: 항등원을 어떤 값으로 초기화 할지 명시
		ex) omp_prive = 0

CUDA Thread ID
	![[Pasted image 20251128233735.png]]
	CUDA는 수많은 스레드를 2단계 계층 구조(그리드와 블록)로 구성하며, 각 스레드는 고유한 위치 정보를 통해 자신이 어떤 작업을 해야 할지 결정함
	1. 블록 ID(blockIdx): 스레드 블록이 전체 그리드 내에서 어디에 위치하는지 나타냄
	   **변수**: blockIdx.x, blockIdx.y, blockIdx.z (각 축의 인덱스)
	2. 스레드 ID(threadIdx): 개별 스레드가 자신이 속한 블록 내에서 어디에 위치하는지 나타냄
	   **변수**: threadIdx.x, threadIdx.y, threadIdx.z (각 축의 인덱스)
	kernel <<<dim3(2, 2), dim3(3, 3)>>> -> 그리드에서 블록(2, 2)의 (3, 3)스레드 호출

CUDA 스레드 블록을 SM으로 매핑하기
	1개의 SM에 여러개의 스레드 블록 존재 가능 -> 다른 블록의 스레드들은 협업 어려움
	블록들 사이에는 기본적으로 동기화 안됨
	블록 안의 스레드끼리는 메모리와 레지스터 공유
	1개의 thread block -> 1개의 SM
	- 그리드는 전체 커널 실행을 나타내는 최상위 개념
	- 하나의 그리드는 GPU 내의 모든 SM에 걸쳐 분할되어 실행됨
	- 하나의 SM이 그리드 전체를 소유하는 것이 아니라, 그리드는 SM들 간에 작업을 분배하는 대상임
	- 스레드 블록과 SM은 1:1로 매핑되어 스케줄링 됨 -> 하나의 SM은 여러 개의 블록을 동시에 실행하거나 대기시킬 수 있음


CUDA 스레드를 Warp에 매핑
	Warp: CUDA GPU의 기본 스케줄링 단위(병렬 컴퓨팅의 최소 실행 단위)
	- 32개 스레드를 warp 단위로 묶어줌
	- warp의 스레드들은 하나의 instruction으로 동작
	2D/3D block: Linear ID = threadIdx.x + (threadIdx.y * blockDim.x) + (threadIdx.z * (blockDim.x * blockDim.y))

SM, Warp, 스레드 블록의 관계
	Block: 하나의 SM을 공유할 스레드들 (반)
	Warp: 실행 유닛 - 한 반에 같이 수업을 들을 학생 (오전, 오후 반)
	SM: warp를 실행하는 하드웨어 엔진 - 하나의 교실 (물리적인 공간)
	.
	스레드 블록 -> SM(할당): 하나의 블록은 하나의 SM에 들어감
	스레드 블록 -> warp(분할): 블록을 32개씩 나눔
	Warp -> SM: 워프를 스케줄링 해서 SM 사용
		하나의 SM은 여러개의 warp를 호출 가능
		스케줄러는 지금 당장 실행할 수 있는 warp를 고름 -> 메모리 레이턴시 숨김
